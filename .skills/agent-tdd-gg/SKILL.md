# 测试驱动开发 (TDD)

## ## 核心理念

**核心原则**：测试应该通过公共接口验证行为，而非实现细节。代码可以完全改变，但测试不应该随之改变。

**好的测试**：是集成式的。它们通过公共 API 走真实的代码路径。它们描述系统**做了什么**，而不是**怎么做的**。好的测试读起来就像规格说明书——“用户可以用有效购物车结账”准确地告诉你存在什么能力。这些测试能在重构中存活，因为它们不关心内部结构。

**坏的测试**：与实现耦合。它们模拟内部协作者、测试私有方法，或通过外部手段验证（比如直接查询数据库而不是使用接口）。危险信号：你重构时测试挂了，但行为并没有改变。如果你重命名的内部函数导致测试失败，那么这些测试测的是实现，不是行为。

参见 [tests.md](tests.md) 了解示例，[mocking.md](mocking.md) 了解模拟指南。

## ## 反模式：水平切片

**不要先写完所有测试，再写所有实现。** 这是“水平切片”——把 RED 当成“写所有测试”，把 GREEN 当成“写所有代码”。这会产生糟糕的测试。

**应该使用垂直切片**：针对一个微小的行为（如一个快乐路径的边界），走完“红-绿-重构”全流程，然后再进入下一个微小的行为。
