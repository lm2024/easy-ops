## 1. 什么是专家团模式

```
传统模式：
┌─────────────────────────────────────┐
│  你：帮我开发用户管理功能            │
│  AI：直接写代码                      │
└─────────────────────────────────────┘

专家团模式：
┌─────────────────────────────────────┐
│  你：用专家团模式开发用户管理功能    │
│                                     │
│  AI 内部：                            │
│  1. 架构师：设计方案                  │
│  2. 后端：按方案实现 Java             │
│  3. 前端：按方案实现 Vue              │
│  4. 测试：写测试用例                  │
│  5. 审查：检查代码质量                │
└─────────────────────────────────────┘
```

**核心价值**：
- ✅ 结构化工作流程（设计 → 开发 → 测试 → 审查）
- ✅ 角色分工明确（每个"专家"专注自己的领域）
- ✅ 代码质量更高（多人协作，互相检查）
- ✅ 可复用（一套 Skill 文件，多种工具使用）

## 3. 核心概念

### 3.1 Skill 文件

```
Skill 文件 = 给 AI 的"角色说明书"
```

**标准结构**：
```
{专家名}/SKILL.md
├── YAML 元数据（开头 3 行）
│   ├── name: 专家名称
│   ├── description: 何时触发
│   └── ---（分隔符）
└── Markdown 正文
    ├── 角色定义
    ├── 工作流
    ├── 输出格式
    └── 协作规则
```

### 3.2 角色分工

| 角色 | 职责 | 输出物 |
|------|------|-------|
| **架构师** | 系统设计、模块划分、API 契约 | 架构方案文档 |
| **后端** | Java 代码实现（Spring Boot） | Controller/Service/Mapper |
| **前端** | Vue 3 页面实现 | View 组件 + API 调用 |
| **测试** | 单元测试 + 集成测试 | JUnit 测试用例 |
| **审查** | 安全/性能/质量检查 | 审查报告 |

### 3.3 协作流程

```
你（发起者）
  ↓
架构师（设计方案）
  ↓
后端（实现 Java 代码）
  ↓
前端（实现 Vue 页面）
  ↓
测试（编写测试用例）
  ↓
审查（质量检查）
  ↓
你（验收）
```

## 5. Skill 文件通用写法

### 5.1 标准模板

```markdown
---
name: expert-{角色名}
description: 何时触发这个专家
---

# {角色名}专家

## 你的角色
[一句话描述职责]

## 背景知识
[项目架构、技术栈、约束条件]

## 你的工作流
1. [步骤 1]
2. [步骤 2]
3. [步骤 3]

## 输出格式
[示例：如何结构化输出]

## 与其他专家的协作
[说明如何与其他角色交互]
```

### 5.2 关键要素

| 要素 | 说明 | 示例 |
|------|------|------|
| **YAML 元数据** | 工具自动扫描的关键信息 | `name: expert-architect` |
| **角色定义** | 一句话概括职责 | "你负责系统架构设计" |
| **背景知识** | 项目架构、技术栈 | "Java 8 + Spring Boot 2.7" |
| **工作流** | 具体执行步骤 | "1. 接收需求 → 2. 设计方案 → 3. 输出文档" |
| **输出格式** | 结构化输出示例 | "API 设计 → 数据模型 → 文件清单" |
| **协作规则** | 如何与其他角色交互 | "方案确定后发送给后端专家" |

### 5.3 常见错误

```
❌ 错误 1：没有 YAML 元数据
   → 工具无法自动识别

❌ 错误 2：目录命名不规范
   → 应该：.codebuddy/skills/expert-architect/
   → 不要：.codebuddy/skills/architect/

❌ 错误 3：角色职责重叠
   → 架构师不写代码，后端不设计架构

❌ 错误 4：缺少项目背景
   → AI 不知道你的技术栈，输出通用方案
```

## 7. 常见问题

### Q1：为什么其他工具不能像 CodeBuddy 那样自动加载 Skill？

```
答：这是工具协议差异。

CodeBuddy：
  - 内置"专家团"协议
  - 自动扫描 {expert-*}.SKILL.md
  - 自动创建多 Agent 协作

其他工具：
  - 没有"专家团"协议
  - 需要你手动提供 Skill 文件
  - 需要在提示词里手动指定角色
```

### Q2：一套 Skill 文件能通用所有工具吗？

```
答：可以，但需要复制到不同目录。

共享方式：
跨工具专家团/
├── claudecode-skill/     ← 复制到 .claude-code/skills/
├── opencode-skill/       ← 复制到 .opencode/skills/
├── cline-skill/          ← 复制到 .cline/skills/
└── ...

每个目录下的 Skill 文件可以完全相同（内容通用）。
```

### Q3：如何判断一个工具是否支持"专家团模式"？

```
判断方法：

1. 查看工具文档，搜索"expert team"或"agent team"
2. 尝试输入"用专家团模式"，看是否自动识别
3. 检查工具是否自动扫描 {expert-*}.SKILL.md

目前只有 CodeBuddy 原生支持。
```

### Q4：如何自定义 Skill 文件？

```
答：参考第 8 章"如何用模型编写 Skill"。

核心步骤：
1. 理解业务需求
2. 用模型（如 Claude/GPT）生成 Skill 文件
3. 根据代码结构调整
4. 测试验证
```
