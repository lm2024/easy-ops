# 跨工具专家团 · 总结文档

> 本文档是整个"跨工具专家团"项目的总结，帮助你快速理解整体架构和使用方法。

---

## 一、项目概述

### 什么是"跨工具专家团"？

```
传统模式：
┌─────────────────────────────────────┐
│  你：帮我开发用户管理功能            │
│  AI：直接写代码（角色混乱）           │
└─────────────────────────────────────┘

专家团模式：
┌─────────────────────────────────────┐
│  你：用专家团模式开发用户管理功能    │
│                                     │
│  AI 内部：                            │
│  1. 架构师：设计方案                  │
│  2. 后端：实现 Java 代码              │
│  3. 前端：实现 Vue 页面              │
│  4. 测试：编写测试用例               │
│  5. 审查：检查代码质量               │
└─────────────────────────────────────┘
```

**核心价值**：
- ✅ 结构化工作流程（设计 → 开发 → 测试 → 审查）
- ✅ 角色分工明确（每个"专家"专注自己的领域）
- ✅ 代码质量更高（多人协作，互相检查）
- ✅ 可复用（一套 Skill 文件，多种工具使用）

## 五、协作模式

### 模式 1：单工具内部专家团（最简单）

```
适合场景：只有一个工具，想获得"专家团"效果

使用方法：
"请用专家团模式帮我开发 XX 功能"

效果：
- AI 自动模拟"架构师/后端/前端"角色
- 单对话上下文，手动切换角色
- 缺少真正的多 Agent 协作"
```

### 模式 2：双工具接力（推荐新手）

```
适合场景：两个工具配合，分工明确

示例：Hermes（设计）+ Claude Code（开发）

第 1 步：Hermes（架构师）
  输出：架构方案 + API 契约

第 2 步：Claude Code（开发）
  输出：Java 代码 + Vue 页面"
```

### 模式 3：多工具接力（高级）

```
适合场景：复杂项目，需要全方位覆盖

示例：Hermes → Codex → Claude Code → CodeBuddy → Cline

第 1 步：Hermes（总指挥）→ 输出架构方案
第 2 步：Codex（架构师）→ 输出详细设计
第 3 步：Claude Code（后端）→ 输出 Java 代码
第 4 步：CodeBuddy（前端）→ 输出 Vue 页面
第 5 步：Cline（测试）→ 输出测试用例"
```

## 六、Skill 文件结构

### 标准结构

```markdown
---
name: expert-{角色名}
description: 何时触发这个专家
---

# {角色名}专家

## 你的角色
[一句话描述职责]

## 背景知识
[项目架构、技术栈、约束条件]

## 你的工作流
1. [步骤 1]
2. [步骤 2]
3. [步骤 3]

## 输出格式
[结构化输出示例]

## 与其他专家的协作
[协作规则]
```

### 关键要素

| 要素 | 说明 | 示例 |
|------|------|------|
| **YAML 元数据** | 工具自动扫描的关键信息 | `name: expert-architect` |
| **角色定义** | 一句话概括职责 | "你负责系统架构设计" |
| **背景知识** | 项目架构、技术栈、约束条件 | "Java 8 + Spring Boot 2.7" |
| **工作流** | 具体执行步骤 | "1. 接收需求 → 2. 设计方案 → 3. 输出文档" |
| **输出格式** | 结构化输出示例 | "API 设计 → 数据模型 → 文件清单" |
| **协作规则** | 如何与其他角色交互 | "方案确定后发送给后端专家" |

## 七、如何自定义 Skill

### 5 步编写法

```
1. 收集项目信息（架构、技术栈、约束）
2. 向模型提问（使用提示词模板）
3. 审查模型输出（检查清单）
4. 测试验证（实际使用）
5. 迭代优化（根据结果调整）"
```

### 向模型提问示例

```
提示词：

"请帮我编写一个 {角色名} 的 Skill 文件。

【项目信息】
- 架构：[描述架构]
- 技术栈：[列出技术栈]
- 约束条件：[列出约束]
- 业务领域：[描述业务]

【角色职责】
[描述这个角色需要做什么]

【输出要求】
请输出完整的 SKILL.md 文件，包含：
1. YAML 元数据
2. 角色定义
3. 背景知识
4. 工作流
5. 输出格式
6. 约束条件"
```

## 八、常见问题

### Q1：Skill 文件没有被加载？

```
解决方案：

1. 检查目录是否正确
   ✅ 正确：.{工具名}/skills/expert-architect/SKILL.md

2. 检查文件名是否正确
   ✅ 正确：SKILL.md（大写）

3. 检查 YAML 元数据
   ✅ 正确：name: expert-architect

4. 重启工具
   → 退出工具，重新打开"
```

### Q2：AI 没有按角色输出？

```
解决方案：

1. 在提示词中强调角色
   "请以后端专家身份，输出 Java 代码"

2. 检查 Skill 文件中的"角色定义"部分

3. 提供具体示例
   "请输出 Controller/Service/Mapper 三层代码"
```

### Q3：如何判断一个工具是否支持"专家团模式"？

```
判断方法：

1. 查看工具文档，搜索"expert team"或"agent team"
2. 尝试输入"用专家团模式"，看是否自动识别
3. 检查工具是否自动扫描 {expert-*}.SKILL.md

目前只有 CodeBuddy 原生支持。"
```
