## 后端内网离线部署（私有化依赖，JDK 8）

### 用户诉求
前端有 node_modules 可打包拷贝内网；问后端 jar 能否同样「工程内私有化」以便内网拷贝、改码、重打包。内网环境 **JDK 8**，必须能在内网运行。

### 结论与做法
- **Maven 没有 node_modules 扁平文件夹**，其「依赖库」是本地仓库（按 `groupId/artifactId/version` 分层 + POM 元数据）。等价做法是把本地仓库私有化进工程。
- 用 `mvn -Dmaven.repo.local=local-repo dependency:go-offline` + `install -DskipTests` 把全部构建依赖与本项目模块导出到 **`backend/local-repo/`**（149M）。
- **已验证离线重打包**：`mvn -o -Dmaven.repo.local=local-repo clean package -DskipTests` → Reactor SUCCESS（common/server/agent），server fat jar 字节码 major=52（Java 8）。
- 交付物：`backend/local-repo/`、`backend/server/target/ops-platform-server-1.0.0-SNAPSHOT.jar`（Java8）、`agent/target/easy-ops-agent-1.0.0-SNAPSHOT.jar`、`backend/start-prod.sh`（Linux 友好，用系统 java）、`backend/build-offline.sh`（离线重打包）、`backend/DEPLOY.md`。**注意：拷贝时必须带 `backend/server/data/`（H2 库），否则数据丢失。**

### 为适配 JDK 8 修复的测试代码 bug（会阻塞 JDK 8 整包构建，但不影响运行）
- 测试误用 Java 9/11 API：`Map.of` / `Path.of`。
- `server/.../AIAnalyzeControllerTest.java`：`Map.of` → `Collections.singletonMap`。
- `server/.../FileControllerTest.java`：`Map.of(...)` → `new HashMap` + `put`。
- `agent/.../FileCommanderTest.java`、`LogCommanderTest.java`：`Path.of` → `java.nio.file.Paths.get`。
- 原因：原 fat jar（6/28）用 JDK 21 编（target 1.8 故字节码仍是 Java8），`Map.of` 在 JDK21 能编；一旦到 JDK 8 即编译失败。原 `start.sh` 写死 `$HOME/.jdk8/Contents/Home`（macOS 路径），内网请用 `start-prod.sh`。

### 坑
- `-Dmaven.repo.local=backend/local-repo` 在 CWD=backend/ 下会被解析成嵌套 `backend/backend/local-repo`；正确相对路径应为 `local-repo`（已从嵌套位置 mv 到正确位置复核）。
- `-DskipTests` 只跳过测试**运行**不跳过**编译**；JDK 8 下测试编译失败仍需先修 `Map.of`/`Path.of` 或用 `-Dmaven.test.skip=true`。

## 前端登录 403（CORS）修复 —— 任意 IP 可访问

### 现象
用户用局域网地址 `http://172.20.10.2:3000` 访问前端，登录 `POST /api/auth/login` 浏览器报 **403 Forbidden**。curl 三种方式（直连 8081 / 代理 localhost:3000 / 代理 172.20.10.2:3000）都返回 200；但 curl 带上 `Origin: http://172.20.10.2:3000` 即 403 → 根因是 **Spring CORS 拒绝**，非登录逻辑 bug。

### 根因
- CORS 白名单写死在 `backend/server/src/main/resources/application.yml` 第 73 行 `cors.allowed-origins: "http://localhost:3000,http://localhost:5173"`。
- 该值**覆盖**了 `WebConfig.java` 的 `@Value("${cors.allowed-origins:默认值}")`，所以改 @Value 默认值对运行无效——必须改 application.yml 本身。
- 浏览器用 Network 地址访问时 Origin=http://172.20.10.2:3000 不在白名单 → Spring `CorsFilter` 返回 403。

### 修复
- 按用户要求「内网任意 IP 都能访问、跨域不能阻碍」：将 `application.yml` 的 `cors.allowed-origins` 改为 `"*"`（放行所有来源；`WebConfig` 用 `allowedOriginPatterns` + `allowCredentials(true)`，pattern `*` 合法，会回显实际 Origin）。
- 同时影响 `WebSocketConfig` 的 `setAllowedOrigins`（`*` 亦允许所有 WebSocket 源）。

### 后端重启要点（本次踩坑）
- 之前 8081 进程被回收后，直接 `java -jar` 启动**失败**：`SecurityConfig.validateSecurityConfig` 校验 `jwt.secret` 来自环境变量 `JWT_SECRET`，缺失则抛 `SECURITY VIOLATION` 启动失败。须 `export JWT_SECRET=...`（start.sh 有默认值 `easy-ops-jwt-secret-key-2024-secure-random`）。
- 端口：dev profile 实际起在 60192（application.yml `port:8081` 但某处覆盖），与前端 Vite 代理目标 `localhost:8081` 不符。启动须强制 `--server.port=8081` 对齐前端代理。
- 正确启动（常驻）：`export JAVA_HOME=corretto-8; export JWT_SECRET=...; exec java -jar server/target/ops-platform-server-1.0.0-SNAPSHOT.jar --server.port=8081`（run_in_background 前台托管，避免 nohup& 被 shell 回收）。JDK 8 保证内网兼容。

## 部署失败（健康检查未通过）根因与修复 ✅

### 现象
用户通过 UI 立即部署 `demo-test-app.jar`(17MB) 到 agent-1：停止旧进程→查找Jar→传输200→启动200→健康检查 **5 次全部 ⏳ 未通过** → ❌ 部署失败。

### 根因（确定，已实证）
- Agent `ProcessController.start()`（backend/agent/.../controller/ProcessController.java）执行生成的 `start.sh`：
  `JAR_NAME=xxx.jar\nnohup java -jar $JAR_NAME > logs/startup.log 2>&1 &`
- **部署目录下没有 `logs/` 目录** → shell 重定向 `> logs/startup.log` 失败 → **java 进程根本没启动**。实证：容器内所有 `startup.log` 全空、无 app 进程、8080 无响应。
- Server 健康检查硬编码 `curl -s --max-time 3 http://127.0.0.1:8080/hello`（DeployController 第 218 行），容器内 app 没起来 → 永远拿不到 `Hello/DEPLOYED` → 5 次失败。

### 修复
- `ProcessController.start()` 改为：`cd deployDir && mkdir -p logs && setsid sh start.sh > /dev/null 2>&1 < /dev/null &`
  - `mkdir -p logs` 保证重定向目标目录存在 → java 真正启动。
  - `setsid` 让 app 彻底脱离 Agent 进程组，避免 Agent 处理完请求后 app 被连带回收（原 `nohup ... &` 在 `Runtime.exec` 子 shell 退出后仍可能随进程组被回收）。
- 重新离线构建 agent jar（JDK8）：`mvn -o -Dmaven.repo.local=local-repo -pl agent -am package -DskipTests`。
- **Docker 镜像无法重建**（本地无 `eclipse-temurin:8-jre` 缓存且无外网）→ 用 `docker cp agent.jar ops-agent-{1,2,3}:/app/agent.jar` + `docker restart` 三个容器，可写层保留新 jar，等价于升级 agent。（内网若自建镜像需把新 agent.jar 烤进镜像；或离线 `docker build` 需先导入基础镜像。）

## 健康检查开关（可配置 + 可关闭）✅

### 用户诉求
用户工程都没有健康地址，「能不能做个开关，暂时不配置也能部署？」→ 实现项目级「部署后健康检查」开关，关闭则跳过健康检查直接判定成功；开启时可配置端口/路径/关键字。

### 改动
- **后端模型**：`common/.../ProjectModel.java` 加 4 字段：`healthCheckEnabled`(Boolean)、`healthCheckPort`(Integer)、`healthCheckPath`(String)、`healthCheckKeyword`(String)。
- **schema 迁移**：`server/src/main/resources/db/schema.sql` 用 `ALTER TABLE project_info ADD COLUMN IF NOT EXISTS health_check_enabled/port/path/keyword`（带默认值 1 / 8080 / '/hello' / 'Hello,DEPLOYED'），对已有数据零破坏，重启即生效。
- **Mapper**：`ProjectMapper.xml` insert/update 增加这 4 列。
- **DeployController**：`publish()` 与 `rollback()` 健康检查改为读项目配置——`hcEnabled = getHealthCheckEnabled()==null || getHealthCheckEnabled()`；关闭则 `healthy=true` 并记 `⏭️ 健康检查已关闭，跳过检查，直接判定成功`；开启则用 `hcPort/hcPath/hcKeyword` 拼 `curl 127.0.0.1:{port}{path}`，关键字按逗号拆分任一包含即健康。
- **前端**：`ProjectFormView.vue` 加「部署后健康检查」`a-switch`（默认开）+ 端口/路径/关键字三输入（v-if 显示）；`formState` 默认 `true/8080/'/hello'/'Hello,DEPLOYED'`；编辑模式 `formState.value = data` 自动回填。`types/index.ts` 的 `ProjectModel` 补 4 字段。
