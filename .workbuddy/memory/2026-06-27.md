### 配置文件管理 Bug 修复 (17:07)
**Bug 1 — 编辑节点下拉不显示**: ConfigManageView.vue 中 `projectNodes` 计算属性使用 `split(',').includes(n.id)` 时 `n.id` 是 Number，`includes` 严格相等，字符串 `"1"` !== 数字 `1`。修复：`split(',').map(s => Number(s.trim()))`

**Bug 2 — 分发 500 ClassCastException**: ConfigMgmtController 中 `(List<Long>) body.get("targetNodeIds")` 强转失败，Jackson 将 JSON 数组 `[1,2,3]` 反序列化为 `List<Integer>`。修复：新增 `toLongList(object)` 工具方法，在 compare 和 distribute 方法中使用。

**前端操作逻辑优化**:
1. 分发弹窗默认只选当前编辑节点（而非全选所有节点）
2. 分发后展示每节点成功/失败结果表格（status + error 详情）
3. 分发失败时显示具体错误信息（catch e.response.data.message）
4. textarea 占位提示更新：明确告知可手动输入内容后分发

**Bug 3 — 读取离线节点报 400 无友好提示 (17:13)**:
- 原因：Docker 未运行导致 3 个节点 status=0，`requireOnlineNode()` 抛 BusinessException(1002)
- 前端 `loadContent()` 只有 try-finally 无 catch，错误被 axios 拦截器吞掉只弹通用错误
- 修复：加 try-catch，`contentError` ref + warning Alert 显示具体错误信息并提示可手动输入

**Bug 4 — 分发 405 Method Not Allowed (17:17)**:
- Docker 容器的 Agent jar 是旧版，只有 `GET /file/config`，没有 `@PostMapping("/config")`
- 修复：`cp agent/target/*.jar agent/docker/` → `docker compose up -d --build` 重建镜像
- 额外修复：AgentClient.extractDataString 不处理 data 为 String 的情况，导致读取返回 `{code=200, data=...}` toString 而非纯净字符串。加 `rawData instanceof String` 判断

**UX 优化 — 内容来源指示 (17:24)**:

**分发重启通知 + 自动刷新快照 (17:35)**:
- 问题1：勾选"重启"后分发，看不到重启是否生效
- 问题2：分发后节点快照表格不自动更新
- 后端修复：`ConfigDistributeService.distribute` 每个节点增加 `restarted`/`restartSuccess`/`restartError` 字段；`restartProject` 移除 try-catch 吞异常
- 前端修复：分发结果表格增加「重启」列（已重启/重启失败/—）；分发后调用 `refreshSnapshotsSilently()` 静默刷新快照

**Bug 5 — 节点快照永远为空 (17:38)**:
- 分发成功后节点快照表格仍然一片空白
- 根因：`getConfigSnapshot` API 返回 `{ configFile, nodes, allSame }` 结构，但 `selectFile()` 中直接 `(res.data || []).map(...)` 把对象当数组用
- `res.data` 是对象（含 nodes 数组），不是数组，`.map()` 方法不存在 → TypeError 被 try-finally 吞掉 → snapshots 永远为空
- 修复：`selectFile()` 改为 `const data: ConfigSnapshotResult = res.data; snapshots.value = data.nodes.map(...)`
- 连带修复：API `getConfigSnapshot` 返回类型从 `Result<NodeConfigSnapshotModel[]>` 改为 `Result<ConfigSnapshotResult>`；types 新增 `ConfigSnapshotResult` 接口

**Bug 6 — 查看快照间歇 400 / 节点状态抖动 (17:44)**:
- 点击节点快照「查看」按钮有时正常有时 400
- 根因1：Agent 心跳间隔 30s > Server 离线阈值 20s → 节点每 30s 离线约 5-10s，在此期间读取配置触发 `requireOnlineNode()` 抛 BusinessException(1002)
- 根因2：axios 错误拦截器对 400 统一弹 `message.error("Request failed with status code 400")`，用户看到的是原始 axios 错误消息而非业务消息
- 修复1：`application.yml` `offline-second` 从 20 改为 90（远大于 Agent 30s 心跳间隔，避免抖动）
- 修复2：`request.ts` 错误拦截器对 400 状态码不弹全局错误提示，替换 `error.message` 为业务消息后 `reject`，由调用方自行处理（如 `ConfigManageView.loadContent()` 已有黄色提示）

**Bug 7 — 日志管理选择节点无显示 (17:51)**:
- LogManageView.vue 第151行与 ConfigManageView 完全相同的 bug：`proj.nodeIds.split(',').map(s => s.trim())` 产生字符串数组，`ids.includes(n.id)` (number) 严格匹配失败
- 修复：`.map(s => Number(s.trim()))`

**Bug 8 — 日志管理日志文件下拉为空 (17:56)**:
- 原因：`LogMgmtService.requireProfile()` 项目无日志配置时抛出 BusinessException(1002)，前端 `watch(nodeId)` 无 try-catch 静默失败
- 后端修复：`requireProfile()` 和 `getProfile()` 中无配置时自动创建默认 profile (logDir=/app/data/logs, mainLogFile=agent.log, timestampRegex/Format 默认值)；修复 `timestamp_regex NOT NULL` 约束缺失导致的 DataIntegrityViolationException
- 前端修复：`watch(nodeId)` 加 try-catch，新增 `logFilesError` ref + warning Alert 提示；新增「日志配置」Drawer

**Bug 10 — 多节点聚合查询无结果 (18:19)**:
- 原因1：后端 `/logs/aggregate` 返回 `data.lines`（条目数组），前端 `fetchAggregate()` 取 `res.data?.list`，永远 undefined
- 原因2：后端条目字段名 `content`，但表格列 `dataIndex: 'line'` 不匹配
- 修复：`fetchAggregate()` 改取 `data.lines`；表格列 `dataIndex: 'content'`；types 新增 `LogAggregateResult`；增加 try-catch 错误提示
- 原因：后端 `/logs/view` 返回 `data.content`（字符串），前端 `fetchLogView()` 取 `res.data.lines`（数组），`undefined` → fallback `[]` → 显示"暂无日志"
- 修复：`fetchLogView()` 改为取 `data.content` 再 `.split('\n')` 分割；types `LogViewResult` 增加 `content`/`logPath` 字段 支持自定义 logDir/mainLogFile/rollingPattern/timestampFormat/maxLineLength；`logFilesLoading` ref 显示加载状态；节点选择器增加 :loading="logFilesLoading"

**Bug 11 — 搜索 500 / 单节点缺少选择器 / 上下文说明不清 (18:27)**:
- 原因1：`LogMgmtController.search()` 中 `(List<Long>) body.get("nodeIds")` Jackson 反序列化为 `List<Integer>`，强转 ClassCastException → 500
- 原因2：搜索 tab 的"单节点"模式下没有节点选择器，只能用聚合 tab 的节点多选框，体验混乱
- 原因3："上下文"输入框无说明，用户不理解含义
- 后端修复：`LogMgmtController` 新增 `toLongList()` 安全转换方法替代强转
- 后端修复：`LogSearchService.fallbackSearch()` 新增 context lines（匹配行上下各 N 行），用 `>>>` 标记匹配行
- 前端修复：搜索 tab 新增 `searchNodeId` 节点选择器（仅 SINGLE 模式显示）；未选节点时 `message.warning` 提示
- 前端修复：搜索结果字段映射修正 — `res.data.hits`（非 matches），字段 `matchedLine`/`lineNo`/`file`（非 line/lineNumber/fileName）
- 前端修复：上下文数字输入框增加 `<a-tooltip>`："匹配行上下各显示 N 行，帮助理解日志发生的背景"
