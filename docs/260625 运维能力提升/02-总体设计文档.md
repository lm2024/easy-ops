# EasyOps 运维能力提升 — 总体设计文档 v1.0

> 基于 [01-需求.md](./01-需求.md) 编写，在现有 Server-Agent 架构上扩展五大能力模块。

---

## 1. 背景与目标

### 1.1 背景

生产环境为**内网老微服务**架构：同一应用多机部署，各节点 `yml` 配置可能不一致；无配置中心、无 ELK/Loki 等日志中间件；运维人员难以解读 JVM/监控指标。

### 1.2 建设目标

| 目标 | 说明 |
|------|------|
| 配置可控 | 跨节点配置对比、差异编辑、批量/单独分发 |
| 日志可查 | 单节点查看 + 同应用多节点聚合，支持关键词搜索 |
| 监控可懂 | 多维指标 + AI 诊断，关键信息自动截取 |
| 知识可沉淀 | 内网 Markdown 知识库，运维-开发协同 |
| 故障可自愈 | 独立自愈模块：自动重启 → 邮件 + 全员页面告警 |

### 1.3 约束

- **纯内网**，不引入配置中心、日志中间件、外部 SaaS
- **JDK 8** + **Spring Boot 2.7.18** + **H2** + **Vue 3**
- 复用现有 Agent HTTP/WebSocket 通道，Server 编排、Agent 执行

### 3.1 复用能力

| 现有组件 | 复用方式 |
|---------|---------|
| `AgentProxyController` | 扩展代理 config/log/shell/metrics |
| `FileController` (Agent) | 增加 config 写入、log grep、tail |
| `AIAnalyzeController` | 扩展为结构化诊断，增加智能截取 |
| `AlarmService` + SMTP | 自愈失败触发邮件 |
| `WebSocket` 基础设施 | 新增 `NotificationHandler`、`LogTailHandler` |
| `project_info` | 扩展 deploy_dir、健康检查、日志/配置路径 |
| `ConfigEditor` / `FileLogView` | 迁移为新页面，旧路由保留兼容跳转 |

### 3.2 废弃/降级

| 组件 | 处理 |
|------|------|
| Agent `AutoRestartDaemon` | 标记 `@Deprecated`，逻辑迁移至 Server `SelfHealScheduler` |
| Server `FileController` 本地读文件 | 改为统一代理 Agent（修复当前未走 Agent 的问题） |
| `MonitorController` 占位数据 | 替换为真实采集 |

## 6. 关键技术决策

### 6.1 配置 diff

- Server 端使用 `java-diff-utils` 生成 unified diff
- 内容哈希 SHA-256 判断「配置是否相同」

### 6.2 老微服务健康探测（无 Actuator）

老项目**不能改代码**，不依赖 Spring Boot Actuator。采用 Agent 侧**零侵入**双层探测：

| 层级 | 手段 |
|------|------|
| L1 进程存活 | `ps` 匹配 `deploy_dir` + `jar_name` |
| L2 HTTP 探针 | 项目级配置 GET/POST/HEAD + URL + 期望状态码 + 可选 body 关键字 |

判定：`RUNNING` = L1 通过且（未配探针或 L2 通过）；自愈与监控共用同一套 `project_health_probe` 配置。

JVM 指标：对 L1 找到的 Java PID 可选执行 `jstat`（不要求应用暴露端点）。

### 6.3 日志聚合（无中间件）

```
用户请求聚合日志
    → Server 并行 HTTP 调用各 Agent GET /file/log/tail
    → 每行解析时间戳（项目级正则）
    → Server 内存归并排序（时间倒序）
    → 返回分页结果
```

搜索：Server 下发 `POST /shell/exec` → Agent 执行 `grep -n "keyword" path`（限制超时 30s）。

### 6.4 AI 上下文预算（128k）

| 区块 | Token 预算 | 内容 |
|------|-----------|------|
| 系统提示 + 结构化监控 | ≤ 8k | CPU/内存/磁盘/JVM/健康状态 JSON |
| 错误日志 | ≤ 50k | ERROR/Exception 行 + 上下文各 5 行 |
| 最近变更 | ≤ 10k | 最近部署记录、最近配置分发 |
| 用户问题 + 回复预留 | ≤ 60k | prompt + max_tokens |

截取优先级：ERROR > WARN > 含 Exception 栈 > 时间最近。

### 6.5 知识库方案

**自研 Wiki 模块**（不引入独立开源服务）：

- 存储：H2 + Server 文件系统 `data/kb/attachments/`
- 编辑：`@kangc/v-md-editor`（Vue3 Markdown）
- 协同：文档编辑锁（30min 超时）+ WebSocket 在线 presence + 评论线程
- 协同：**方案 A**（编辑锁 + 在线人数 + 评论/聊天）
- 图片：内网截图/上传，文内展示；**不支持 PDF**；支持文档 MD/ZIP 下载
- 参考交互：Outline / GitHub Wiki，不引入其运行时依赖

### 6.6 部署脚本

部署时 Server **自动生成**统一 `start.sh` / `stop.sh` 下发 Agent，不依赖各环境原有脚本名（如 `startTm.sh`）。

### 6.7 实时通知

- 新 WebSocket：`/ws/notification`（用户 Token 鉴权）
- 消息类型：`ALERT`（需手动关闭）、`INFO`、`SELF_HEAL`
- 离线用户登录后 `GET /notifications/unread` 拉取未关闭的 `ALERT`

## 8. 生产目录映射示例

以 tm 服务为例，项目配置：

```yaml
deploy_dir: /home/stms/tm-server
jar_name: tm.jar
# 部署时平台自动生成 start.sh / stop.sh，无需关心原 startTm.sh
config_paths:
  - /home/stms/tm-server/config/application.yml   # 仅一份生产配置
log_profile:
  main_log: /home/stms/tm-server/logs/app.log
  pattern: "^\\d{4}-\\d{2}-\\d{2} \\d{2}:\\d{2}:\\d{2}"
health_probe:
  enabled: true
  method: GET
  url: http://127.0.0.1:8080/hello
  expected_status: 200
  timeout_ms: 3000
```

## 9. 安全要点

- 配置/日志路径必须在项目 `deploy_dir` 白名单内（SEC-007 扩展）
- 分发/编辑记录写入 `operation_log` + `file_access_log`
- 知识库按 `user_project_relation` 控制可见范围（可选）
- 通知 WebSocket 复用 `WebSocketAuthInterceptor` JWT 校验
- AI 诊断不传 Token/密码等敏感配置项（YAML 脱敏）
