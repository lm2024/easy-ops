### 健康探测方案（老微服务专用，已确认）

**原则：零侵入业务代码，Agent 侧完成所有探测。**

| 层级 | 手段 | 说明 |
|------|------|------|
| L1 进程存活（必开） | `ps` 匹配 `deploy_dir` + `jar_name` | 不依赖 PID 文件，适配各厂商脚本 |
| L2 HTTP 探针（推荐开启） | 项目配置 URL + Method + 期望状态码 | 适配 `/hello`、`/api/status` 等老接口 |
| L3 主机资源 | Agent `/sys/info` | CPU/内存/磁盘，与应用无关 |
| L4 JVM（可选） | 对 L1 找到的 Java PID 执行 `jstat` | 不要求应用暴露端点 |

**HTTP 探针推荐配置（每项目独立）：**

```yaml
health_probe:
  enabled: true
  method: GET          # GET | POST | HEAD
  url: http://127.0.0.1:8080/hello
  headers: {}          # 可选，如 Authorization
  body: ""             # POST 时填写
  expected_status: 200 # 期望 HTTP 状态码
  body_contains: ""    # 可选，响应体需包含的关键字
  timeout_ms: 3000
```

**存活判定逻辑：**

```
RUNNING  = L1 进程存在 AND (未启用 L2 OR L2 探针通过)
DOWN     = L1 进程不存在 OR (启用 L2 且探针失败)
DEGRADED = L1 存在但 L2 超时/非预期状态码（进程在但服务不可用）
```

**为何不默认 POST：** 老接口多为 GET 健康检查；POST 作为可选项，用于需带 body 的少数场景。平台表单提供 Method 下拉即可。
