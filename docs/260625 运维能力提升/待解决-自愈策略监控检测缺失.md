# 待解决：自愈策略监控检测缺失

> 状态：⚠️ 待解决 | 创建日期：2026-06-27

## 问题描述

当前自愈策略模块的故障检测手段过于简陋，无法有效发现服务异常，导致"自愈"能力形同虚设。

核心矛盾：**连服务挂了都检测不到，谈何自愈？**

## 四个设计缺口

### 1. 检测手段单一：只查进程，不查服务

| 当前做法 | 问题 |
|----------|------|
| Agent 执行 `ps aux \| grep` 检查 Java 进程 | 进程存活 ≠ 服务正常 |

以下场景全部漏检：
- Java 进程没死，但 HTTP 返回 503
- 线程池满载，请求排队超时
- 死锁导致接口无响应
- 内存溢出 GC 频繁，响应雪崩
- 端口绑定但业务逻辑崩溃

### 2. checkIntervalSec 未生效

策略编辑页可以配置「检查间隔」，但调度器 `SelfHealScheduler` 中写死 30 秒轮询，策略中的 `checkIntervalSec` 字段完全被忽略。用户看到配置项但不起作用。

- 文件：`SelfHealScheduler.java`
- 需改：读取策略的 `checkIntervalSec`，为每个策略使用独立调度/间隔

### 3. 策略列表无监控可见性

前端策略列表只显示策略名、动作、节点数等静态配置，用户完全不知道：
- 这个策略正在监控哪些节点？
- 上次检测是什么时候？
- 检测结果是什么？成功还是失败？
- 策略是否真的在跑？

### 4. MonitorCollector 的 HTTP 健康探针未对接

`MonitorCollector` 模块已经实现了一套完整的 HTTP 健康探针能力：

- `HttpHealthProber.java`：支持 GET/POST/HEAD 请求 + 期望状态码校验
- `project_health_probe` 表：已存储健康检查结果
- `MonitorScheduler`：定时采集健康状态

但 `SelfHealService` **完全没有对接这套探针**，两套系统各自为战。

## 改造方案（待实施）

### 方案：双路径健康检测

在 `SelfHealService` 中集成 HTTP 健康探针，策略增加「健康检查配置」：

```
策略检测流程：
  ├── L1 进程存活检测（原有 ps aux | grep）
  │     └── 进程不可见 → 判定 DOWN，触发自愈
  └── L2 HTTP 健康探针（新增，复用 HttpHealthProber）
        ├── 策略配置：健康检查 URL、请求方法(GET/POST/HEAD)、期望状态码
        ├── 探针失败 → 判定 DOWN，触发自愈
        └── 探针成功 → 判定 OK
```

### 涉及改动

| 模块 | 文件 | 改动 |
|------|------|------|
| 数据模型 | `self_heal_policy` 表 | 新增 `probe_url`, `probe_method`, `probe_expected_status` |
| 后端 Service | `SelfHealService.java` | `checkAll()` 中集成 HTTP 健康探针，替代纯进程检测 |
| 后端 Scheduler | `SelfHealScheduler.java` | 读取 `checkIntervalSec`，按策略独立调度 |
| 后端 Controller | `SelfHealController.java` | 新增策略状态查询接口（上次检测时间、结果） |
| 前端 API | `selfheal.ts` | 新增状态查询 API |
| 前端 UI | `SelfHealPolicyView.vue` | 列表增加「监控节点数/最近检测/状态」列；编辑增加健康检查配置表单 |

### 依赖关系

- `HttpHealthProber` 已在 Agent 端实现（`backend/agent/src/main/java/com/ops/agent/process/HttpHealthProber.java`）
- `project_health_probe` 表已存在，结构可用
- Server → Agent 的 HTTP 调用链路已通（`AgentProxyController`）

## 影响范围

- 自愈策略在未改造前仅能检测「进程彻底消失」这一种故障，假死/服务降级/503 等常见故障均无法触发自愈
- 不影响其他模块（告警、部署、日志管理等功能独立）
