# EasyOps 纯内网生产部署指南（前端 + 后端）

> 适用场景：无外网的纯内网环境。所有依赖包（Maven 离线仓库 local-repo、前端 node_modules）已提前下载就绪。
> 目标：照着本文档操作即可把 前端 + Server + Agent 部署起来并跑通。

---

## 0. 一句话结论 / 最简架构

最省事的拓扑（单台服务器即可跑全套）：

```
[浏览器] ──http/https──> [Nginx :80] ──代理 /api、/ws──> [Server :8081]
                                               [Agent :2123]（可选，纳管本机或其他机器）
```

- **Server**：后端主服务，端口 8081，上下文 `/api`，数据全在它的 `./data` 目录。
- **Agent**：部署在被纳管的目标机上，端口 2123。Server 通过它下发 jar、启停进程、采集日志。
- **Nginx**：给用户访问前端页面，并把 `/api` 和 `/ws` 转发给 Server。
- **前端**：`npm run build` 生成的静态文件（`dist/`），由 Nginx 托管。前端代码里 API 地址是相对路径 `/api`，**不需要改代码就能换主机**，只要 Nginx 代理对即可。

> 关于你问的「Agent 到底用不用 Token」：见第 6 节。结论先说——**Agent 不配 Token 会拒绝启动（代码写死），但配一个任意字符串即可，一行环境变量搞定，并不复杂。**

## 1. 目录规划（你最关心的部分）

### 1.1 推荐的生产目录结构

在一台 Linux 服务器上，建议这样放（路径随意，这里以 `/opt/easyops` 为例）：

```
/opt/easyops/
├── server/                         # Server 安装目录（启动 jar 的工作目录）
│   ├── ops-platform-server-1.0.0-SNAPSHOT.jar
│   ├── application.yml             # 外部配置文件（可选覆盖，见第 3 节）
│   ├── start.sh                    # 启动脚本（设 JWT_SECRET 等环境变量）
│   ├── backend.log                 # 启动输出日志
│   └── data/                       # ★ server.path 默认 = ./data，自动生成，务必可写
│       ├── ops.mv.db               #     H2 数据库文件（自动初始化，无需手动建库）
│       ├── ops.trace.db            #     H2 跟踪文件
│       ├── logs/server.log         #     运行日志（logging.file.name）
│       └── versions/               #     ★ 上传目录 + 分发源目录
│           └── <projectId>/        #         每个项目一个子目录
│               └── <version>/      #             每个版本一个子目录
│                   └── xxx.jar     #             你通过界面上传的版本包
│
├── agent/                          # Agent 安装目录（仅当你要纳管这台机器时）
│   ├── easy-ops-agent-1.0.0-SNAPSHOT.jar
│   ├── application.yml             # 可选
│   ├── data/                       # Agent 日志目录（./data/logs/agent.log，相对工作目录）
│   │   └── logs/agent.log
│   └── app-data/                   # ★ AGENT_DATA_PATH，部署目标目录（jar 实际运行处）
│       └── <projectId>/...         #     Server 下发的 jar + start.sh/stop.sh 落这里
│
└── frontend/                       # 前端
    └── dist/                       # ★ npm run build 产物，由 Nginx 托管
        ├── index.html
        └── assets/...
```

### 1.2 三类「目录」对应关系（重点）

| 你关心的概念 | 实际位置 | 说明 |
|------|----------|------|
| **上传目录** | Server 的 `./data/versions/<projectId>/<version>/` | 你在「版本包管理」界面上传的 jar 存这里（由 `server.path` 决定，默认 `./data`）。 |
| **分发目录 / 源目录** | 同上 `./data/versions/...` | 部署时 Server 把这个 jar 通过 HTTP 传给 Agent。 |
| **部署运行目录** | Agent 的 `AGENT_DATA_PATH`（默认 `/app/data`，建议改为 `/opt/easyops/agent/app-data`） | jar + start.sh/stop.sh 真正运行的地方。若项目未单独配置 `deployDir`，默认就用这个目录。 |
| **配置文件目录** | jar 同级的 `application.yml`（外部） | 见第 3 / 第 4 节。不写也行，靠环境变量覆盖。 |
| **日志目录** | Server: `./data/logs/`；Agent: `./data/logs/`（相对各自工作目录） | 排错看这里。 |
| **数据库目录** | Server: `./data/ops.mv.db` | H2 文件库，自动建表，无需手动初始化。 |

> 关键提醒：Server 的 `server.path` 是**相对路径 `./data`**，等于「启动 jar 时所在目录下的 data」。所以**一定要在 `/opt/easyops/server/` 目录里启动 jar**，否则数据会散落到你执行命令的目录。Agent 同理（日志在 `./data/logs`，相对启动目录）。

## 2. 前置条件（内网机器需自带）

- **JRE 8+**：后端 jar 编译目标是 Java 1.8，运行需要 JRE 8 或以上（推荐 OpenJDK 8 / 11 / 17 均可）。
- **Nginx**：任意版本（1.18+ 即可），用于托管前端 + 反向代理。内网机器若没有，需提前把 nginx 二进制/rpm 拷进去。
- （可选）**systemd**：用于把 Server/Agent 注册成服务，开机自启。本文给示例。
- 你已完成的：后端离线依赖包（local-repo）、前端 node_modules 已就位 → 说明**构建环境已具备**。

## 3. 后端 Server 部署

### 3.1 构建（若已有 jar 可跳过）

在能联网/已有离线仓库的构建机上：

```bash
cd easy-ops/backend
# 离线构建（指向你下载好的 local-repo）：
mvn -o -Dmaven.repo.local=local-repo clean package -DskipTests
# 产物：
#   server/target/ops-platform-server-1.0.0-SNAPSHOT.jar
#   agent/target/easy-ops-agent-1.0.0-SNAPSHOT.jar
```

> 把这两个 jar 拷到生产机的 `/opt/easyops/server/` 和 `/opt/easyops/agent/`。

### 3.2 配置文件（application.yml）

Server 默认配置已可用，**生产只需关注少数几项**。两种方式任选：

**方式 A（推荐，环境变量，不动文件）**：启动脚本里 export 即可，见 3.3。
**方式 B（外部 yml 覆盖）**：在 jar 同级放 `application.yml`，以下是最简生产版：

```yaml
server:
  port: 8081
  servlet:
    context-path: /api
  path: ./data                 # 数据根目录，保持 ./data（启动目录下的 data）
  heart-second: 5
  offline-second: 90

spring:
  datasource:
    driver-class-name: org.h2.Driver
    url: jdbc:h2:./data/ops;MODE=MySQL
    username: sa
    password:
  sql:
    init:
      mode: always

jwt:
  secret: ${JWT_SECRET:change-me-please}   # 生产务必设一个随机串（见 3.3）

cors:
  allowed-origins: "*"        # 内网放宽；若想收紧改成前端地址清单

ai:
  apiKey: ${AI_API_KEY:}      # 不用 AI 功能可留空
```

> 不写外部 yml 也完全能跑——所有项都有默认值，缺的只是 `JWT_SECRET`（不设为空也能启动，但建议设）。

### 3.3 启动 Server

`/opt/easyops/server/start.sh`：

```bash
#!/bin/bash
set -e
cd "$(dirname "$0")"                      # 关键：切到 jar 所在目录，./data 才正确
export JWT_SECRET="${JWT_SECRET:-$(head -c 32 /dev/urandom | base64)}"  # 随机密钥，重启会变（见下方说明）
nohup java -jar ops-platform-server-1.0.0-SNAPSHOT.jar > backend.log 2>&1 &
echo "Server PID: $!"
```

> ⚠️ `JWT_SECRET` 若每次启动随机，已登录用户的 Token 会在 Server 重启后失效（需重新登录），不影响功能。想要重启不掉登录，就把 `JWT_SECRET` 设成**固定值**（写死或放进环境文件）。

启动后验证：

```bash
curl http://127.0.0.1:8081/api/auth/login -X POST \
  -H 'Content-Type: application/json' \
  -d '{"username":"admin","password":"admin123"}'
# 返回带 token 的 JSON 即成功（默认管理员 admin / admin123）
```

### 3.4 注册成 systemd 服务（可选，开机自启）

`/etc/systemd/system/easyops-server.service`：

```ini
[Unit]
Description=EasyOps Server
After=network.target

[Service]
WorkingDirectory=/opt/easyops/server
Environment=JWT_SECRET=你的固定密钥
ExecStart=/usr/bin/java -jar /opt/easyops/server/ops-platform-server-1.0.0-SNAPSHOT.jar
SuccessExitStatus=143
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

```bash
systemctl daemon-reload
systemctl enable --now easyops-server
journalctl -u easyops-server -f   # 看日志
```

## 4. 后端 Agent 部署

### 4.1 配置文件（application.yml）

Agent 同样可用默认配置，**生产通过环境变量覆盖最省事**。外部 yml 最简版：

```yaml
server:
  port: 2123
  servlet:
    context-path: /api
agent:
  server-url: http://<SERVER_IP>:8081/api   # ★ 指向 Server 地址（内网 IP）
  token: ${AGENT_TOKEN:}                    # ★ 见第 6 节，必须非空
  node-name: ${AGENT_NODE_NAME:node-1}      # 节点显示名
  data-path: /opt/easyops/agent/app-data    # ★ 部署运行目录（jar 落地处）
  host-ip: ${AGENT_HOST_IP:}                # 注册到 Server 的 IP（建议填内网 IP）
  host-port: ${AGENT_HOST_PORT:2123}
logging:
  file:
    name: ./data/logs/agent.log             # 相对启动目录
```

### 4.2 启动 Agent

`/opt/easyops/agent/start.sh`：

```bash
#!/bin/bash
set -e
cd "$(dirname "$0")"
export AGENT_TOKEN="${AGENT_TOKEN:-easyops123}"            # ★ 任意非空串即可
export AGENT_SERVER_URL="http://<SERVER_IP>:8081/api"      # ★ 改成 Server 内网地址
export AGENT_NODE_NAME="${AGENT_NODE_NAME:-node-1}"
export AGENT_DATA_PATH="/opt/easyops/agent/app-data"
nohup java -jar easy-ops-agent-1.0.0-SNAPSHOT.jar > agent.log 2>&1 &
echo "Agent PID: $!"
```

> `AGENT_DATA_PATH` 通过环境变量 `agent.data-path` 注入（Spring 会把 `AGENT_DATA_PATH` 映射成 `agent.data-path`）。

启动后验证：看 Agent 日志出现 `Agent token loaded` 且 Server 的「节点列表」里出现该节点（自动注册，见第 6 节）。

### 4.3 多 Agent（纳管多台机器）

每台被纳管机器都装一份 Agent，只需改 `AGENT_TOKEN`（可与 Server 侧约定一致，也可各不相同）、`AGENT_NODE_NAME`、`AGENT_SERVER_URL`、`AGENT_HOST_IP` 指向各自内网 IP。Docker 方式见仓库 `backend/docker-compose.yml`（3 实例示例）。

## 5. 前端部署（Nginx 托管）

### 5.1 构建（若已有 dist 可跳过）

```bash
cd easy-ops/frontend
npm run build          # 等价于 vue-tsc -b && vite build → 产物在 dist/
# 若类型检查(vue-tsc)报错阻断构建，可绕过类型检查直接打包：
# npx vite build
```

把 `dist/` 整个目录拷到生产机 `/opt/easyops/frontend/dist/`。

### 5.2 Nginx 配置

`/etc/nginx/conf.d/easyops.conf`（或包含进主配置）：

```nginx
server {
    listen 80;
    server_name _;

    # 前端静态资源
    root /opt/easyops/frontend/dist;
    index index.html;
    try_files $uri $uri/ /index.html;   # 支持前端路由（history 模式）

    # 后端 API 代理
    location /api/ {
        proxy_pass http://127.0.0.1:8081/api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    # WebSocket（实时终端 / 监控 / 部署进度）—— 必须，否则控制台和监控无数据
    location /api/ws/ {
        proxy_pass http://127.0.0.1:8081/api/ws/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_read_timeout 3600s;
    }
}
```

```bash
nginx -t && systemctl reload nginx
```

浏览器访问 `http://<SERVER_IP>/` 即可看到登录页，用 `admin / admin123` 登录。

> 前端 API 地址是相对路径 `/api`，因此**换服务器 IP 不用重新打包前端**，只要 Nginx 代理对就行。

## 6. 关于 Token：到底用不用？为什么这么复杂？能不能不用？

你问得很好，这里把真相讲清楚。

### 6.1 事实

- **Agent 端**：`HeartbeatDaemon` 和 `AutoRestartDaemon` 启动时都会检查 `AGENT_TOKEN` 是否为空，**为空就直接抛异常、进程退出**。这是写死在代码里的强制校验，所以在「不改代码」的前提下：**不配 Token，Agent 起不来**。
- **Server 端**：`/api/nodes/heartbeat` 接口要求请求头带非空 token，并**按 token 自动注册节点**（token↔节点绑定）。也就是说——你不需要在界面上提前建节点，Agent 第一次心跳带着 token 来，Server 就自动登记了。

### 6.2 所以「简单部署」其实很简单

Token 在这里**只是个你我约定的共享口令**，不是什么复杂体系。生产上你只要：

```bash
export AGENT_TOKEN=easyops123     # 任意字符串都行
```

Agent 就能启动，Server 自动把它登记成节点。**一行环境变量，完事。** 并不需要生成密钥、不需要证书、不需要提前在界面登记。所谓「复杂」，其实只是「必须有一个非空字符串」这一条限制。

### 6.3 如果就是不想配 Token（彻底去掉）

改 `backend/agent/src/main/java/com/ops/agent/daemon/` 下两个文件，删掉那段非空校验即可：

- `HeartbeatDaemon.java` 的 `run()` 中：
  ```java
  if (agentToken == null || agentToken.trim().isEmpty()) {
      throw new IllegalStateException("SECURITY VIOLATION: AGENT_TOKEN is not configured ...");
  }
  ```
- `AutoRestartDaemon.java` 的 `run()` 中同样那段。

把 `throw ...` 整段删掉（或改成仅打 warn 日志），重新 `mvn package` 打 agent jar，就能零 Token 启动。**注意**：这等于关闭了 Agent 鉴权，纯内网、且网络隔离良好时可这么干，但不推荐暴露在可达网络。

> 建议：先用 6.2 的「一行环境变量」方案，最省事也最安全。

## 7. 分步操作清单（照做即可）

1. **准备机器**：Server 机装 JRE8+、Nginx；每台被纳管机装 JRE8+。
2. **构建/拷贝产物**：把 `ops-platform-server-...jar`、`easy-ops-agent-...jar`、`dist/` 分别放到对应目录（见第 1.1）。
3. **建目录**：`mkdir -p /opt/easyops/server/data/logs /opt/easyops/agent/data/logs /opt/easyops/agent/app-data /opt/easyops/frontend`。
4. **起 Server**：按 3.3 启动；`curl` 登录接口验证返回 token。
5. **起 Agent**：按 4.2 启动（`AGENT_SERVER_URL` 指向 Server 内网 IP，`AGENT_TOKEN` 给个值）；到 Server 界面「节点列表」确认节点在线。
6. **配 Nginx**：按 5.2 写好配置并 `nginx -t && reload`。
7. **浏览器访问** `http://<SERVER_IP>/` → 登录 `admin/admin123`。
8. **验证部署闭环**：在「项目管理」建项目 → 「版本包管理」上传一个 jar → 「部署」选节点发布 → 看部署进度与终端。

## 8. 已知坑 / 注意事项（务必看）

1. **Agent 日志路径是相对启动目录**：`logging.file.name: ./data/logs/agent.log` 是相对路径，务必在 `/opt/easyops/agent/` 目录内启动，否则日志写到别处。
2. **Server 的 `server.path=./data` 也是相对路径**：务必在 server 目录内启动 jar，数据/版本包才会落在预期位置。
3. **知识库协同编辑的 WebSocket 是硬编码 localhost**：前端 `src/composables/useCollab.ts` 写死了 `ws://localhost:8081/api/ws/kb-collab`。这意味着**知识库实时协同编辑只有在浏览器跑在 Server 本机时才正常**；从其他机器访问会连不上。
   - 临时绕开：协同编辑非核心功能，不影响部署/监控/终端。
   - 彻底修：把该行改成 `${location.protocol}//${location.host}/api/ws/kb-collab` 后重新 `npm run build`。
4. **`JWT_SECRET` 随机导致重启掉登录**：见 3.3，想要稳定登录设固定值。
5. **H2 数据库文件锁**：Server 同时只能起一个实例；迁移时连同 `./data/ops.mv.db` 一起拷。
6. **离线仓库与 Maven 版本相关**：若你换用与构建时不同的 Maven 大版本做 `mvn package`，需用该版本重新联网补全依赖（未固定版本的默认生命周期插件会随 Maven 版本变化）。本次已用 Maven 3.5.0 / 3.9.16 双向验证离线构建通过。
7. **Agent 默认 `data-path=/app/data`**：Docker 场景下合理；裸机部署请改成你实际有写权限的目录（如 `/opt/easyops/agent/app-data`），否则部署 jar 时可能因权限/路径不对失败。

## 9. 端口与防火墙速查

| 组件 | 端口 | 用途 | 谁访问 |
|------|------|------|--------|
| Nginx | 80 | 前端页面 + 代理 /api、/ws | 用户浏览器 |
| Server | 8081 | 后端 API（上下文 /api） | Nginx（内网） |
| Agent | 2123 | Agent API（上下文 /api） | Server（内网） |
| Agent 托管应用 | 自定义（如 8080） | 被部署业务自身端口 | 业务调用方 |

内网只需保证：浏览器→Nginx:80、Nginx→Server:8081、Server→Agent:2123 通畅。
