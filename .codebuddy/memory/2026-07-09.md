## 后端离线仓库核查（local-repo 完整性）
- 位置：`easy-ops/backend/local-repo`（Maven 本地仓库，离线包）。共约 360 jar、699 pom。

- 处理：用 `mvn -Dmaven.repo.local=local-repo dependency:go-offline`（联网）补全，仅有此一个项目依赖缺失（其余 maven-repository-metadata 是 go-offline 插件自身依赖）。
- 验证：补全后 `mvn -o package -DskipTests` 纯离线构建 BUILD SUCCESS，产出 server/agent/common 三个 jar。当前离线包已完整。
- 说明：环境有网络（Maven Central 可达），本地 Maven 为 3.9.16（JDK 17 运行；项目 target 1.8 可正常编译）。

## 重要修正：离线仓库与 Maven 版本强相关
- 最初只验证 Maven 3.9.16 离线通过，但用户实际环境是 **Maven 3.5.0**，两者默认生命周期插件版本不同（pom 只固定 maven-compiler-plugin:3.11.0，surefire/jar/resources/clean/install/deploy 未固定）。
- 用 Maven 3.5.0 纯离线构建 **失败**：先缺 `maven-resources-plugin:2.6`，补齐后又缺其传递依赖 `maven-profile:2.0.6`、`maven-repository-metadata:2.0.6`、`maven-plugin-registry:2.0.6`、`classworlds:1.1-alpha-2`，以及 surefire 2.12.4 / maven-archiver 2.5 / plexus-archiver 2.1 等。
- 处理：用 Maven 3.5.0 联网 `package` 把 3.5.0 所需的旧版插件及依赖补齐进 `local-repo`（Maven 3.5.0 二进制已下载到 `C:\Users\Li\Downloads\apps\open-source\maven\apache-maven-3.5.0`）。
- 结论：现 `local-repo` 同时支持 **Maven 3.5.0 与 3.9.16** 纯离线 `mvn -o package -DskipTests` 构建（均 BUILD SUCCESS）。
- 经验：**离线仓库完整性依赖构建所用的 Maven 版本**，未固定版本的默认生命周期插件会解析成该 Maven 自带的版本；换 Maven 大版本必须重新用该版本 `go-offline`/`package` 联网补全。

## 编写生产部署文档
- 产出：`docs/260709-生产部署/生产部署指南.md`（纯内网前后端部署，plain-text markdown）。
- 关键事实（来自代码核对）：
  - Server 数据根 `server.path` 默认 `./data`（相对启动目录）→ 含 H2 库 `ops.mv.db`、日志、版本包 `versions/<projectId>/<version>/`。必须在 server 目录内启动 jar。
  - Agent 日志 `./data/logs/agent.log` 也是相对启动目录；`data-path` 默认 `/app/data`（裸机建议改 AGENT_DATA_PATH）。
  - **Agent 强制要求 AGENT_TOKEN**：`HeartbeatDaemon`+`AutoRestartDaemon` 启动即检查，空则抛异常退出（写死）。Server 心跳按 token 自动注册节点。结论：配任意非空串即可，一行环境变量，并不复杂；想彻底去掉需改这两处 daemon 的校验代码。
  - 前端 API 用相对 `/api`（request.ts baseURL），生产靠 Nginx 反代 `/api` 与 `/api/ws/`，换 IP 不需重新打包。
  - **坑**：`src/composables/useCollab.ts` 硬编码 `ws://localhost:8081/api/ws/kb-collab`，知识库协同编辑仅在本机浏览器正常；其余功能不受影响。
  - 推荐架构：Nginx(:80) → Server(:8081) + Agent(:2123)；前端 dist 由 Nginx 托管。

## 前端 Nginx 部署包（frontend/nginx/）
- 用户要求：在 frontend 工程下放 nginx，配置好注释，拷贝到内网 ARM64 Linux 直接运行。
- 关键事实：**nginx 官方不提供 ARM64 预编译二进制；nginx 1.26.3 的 configure 不支持交叉编译**（auto/ 里无 NGX_CROSS，autotest 必须运行测试程序取 int/long/指针大小，x86 上跑 arm 二进制必失败）。本机是 Windows x86，Docker 未运行，QEMU binfmt 在 WSL 注册不稳定 → 无法在本机产出可验证的 ARM64 二进制。
- 交付：`conf.d/nginx.conf`（带注释，已用 Windows 版 nginx 1.26.3 `nginx -t` 校验语法通过）+ `conf.d/mime.types` + `start.sh`/`stop.sh`/`reload.sh`/`status.sh` + `build-nginx.sh`（在 ARM64 目标机离线静态编译 nginx，自带 src/ 下 nginx-1.26.3 / pcre-8.45 / zlib-1.3.1 源码 tarball）+ `README.md`。
- 运行方式：拷贝整个 `frontend/` 到 ARM64 Linux → `bash build-nginx.sh`（首次，需 gcc/make，~1-2 分钟）→ `bash start.sh`。编译出的 `sbin/nginx` 静态链入 pcre+zlib，仅依赖 glibc，之后整个 nginx/ 目录可再拷到任意 ARM64 Linux 直接 `start.sh`（无需重编）。
- 配置修改点：仅当 Server 不在本机/端口非 8081 时改 `conf.d/nginx.conf` 里两处 `proxy_pass http://127.0.0.1:8081`；或改 `listen 80;`。其余不动。路径均相对 nginx 目录（start.sh 用 `-p` 实现），可移植。
- 说明：`_crossbuild.sh` 是开发机交叉编译尝试脚本（在本机不适用，普通部署忽略）；`src/` 下已有解压后的源码目录（build-nginx.sh 会直接使用，不重复解包）。
