## 1. 什么是跨工具协作

```
传统模式（单工具）：
┌─────────────────────────────────────┐
│  你：帮我开发用户管理功能            │
│  AI：直接写代码（角色混乱）           │
└─────────────────────────────────────┘

跨工具协作模式：
┌─────────────────────────────────────┐
│  你：用专家团模式开发用户管理功能    │
│                                     │
│  Hermes（总指挥）                     │
│    ↓ 设计方案                        │
│  Codex（架构师）                     │
│    ↓ 输出架构方案                    │
│  Claude Code（开发）                 │
│    ↓ 实现 Java 代码                  │
│  CodeBuddy（前端）                   │
│    ↓ 实现 Vue 页面                   │
│  Cline（测试）                       │
│    ↓ 写测试用例                      │
│  Continue（审查）                    │
│    ↓ 检查代码质量                    │
└─────────────────────────────────────┘
```

**核心价值**：
- ✅ 每个工具发挥最强能力
- ✅ 角色分工明确，不混乱
- ✅ 代码质量更高，覆盖全面

## 2. 常见协作模式

### 模式 1：单工具内部专家团（最简单）

```
适合场景：只有一个工具，想获得"专家团"效果

使用方法：
"请用专家团模式帮我开发 XX 功能"

效果：
- AI 自动模拟"架构师/后端/前端"角色
- 单对话上下文，手动切换角色
- 缺少真正的多 Agent 协作
```

### 模式 2：双工具接力（推荐新手）

```
适合场景：两个工具配合，分工明确

示例：Hermes + Claude Code

第 1 步：Hermes（设计）
  你："请用专家团模式，以架构师身份帮我设计用户管理功能"
  输出：架构方案 + API 契约

第 2 步：Claude Code（开发）
  你："请按照以下架构方案，实现用户管理功能：
      [粘贴 Hermes 输出的方案]
      请以后端/前端专家身份实现代码"
  输出：Java 代码 + Vue 页面
```

### 模式 3：多工具接力（高级）

```
适合场景：复杂项目，需要全方位覆盖

示例：Hermes → Codex → Claude Code → CodeBuddy → Cline

第 1 步：Hermes（总指挥）
  你："请担任总指挥，协调以下工作：
      1. 以架构师身份设计系统架构
      2. 输出 API 契约
      3. 定义数据模型"
  输出：完整架构方案

第 2 步：Codex（架构师）
  你："请根据以下需求，以架构师身份完善设计方案：
      [粘贴 Hermes 的方案]
      请补充：数据库设计、模块边界、接口契约"
  输出：详细架构文档

第 3 步：Claude Code（开发）
  你："请根据以下架构方案，实现后端 Java 代码：
      [粘贴 Codex 的方案]
      请输出：Controller/Service/Mapper 代码"
  输出：Java 代码

第 4 步：CodeBuddy（前端）
  你："请根据以下 API 契约，实现前端 Vue 页面：
      [粘贴 API 契约]
      请输出：View 组件 + API 调用"
  输出：Vue 页面

第 5 步：Cline（测试）
  你："请根据以下需求，编写测试用例：
      [粘贴需求文档]
      请输出：JUnit 测试用例 + 前端测试"
  输出：测试用例
```

### 模式 4：全工具竞赛（娱乐）

```
适合场景：对比不同工具的能力

使用方法：
"请用以下工具竞赛，看谁实现得最好：
 1. Claude Code
 2. CodeBuddy
 3. Opencode
 4. Cline"

每个工具独立实现同一功能，你验收最佳方案。
```

## 3. 实战案例 1：Hermes 当总指挥

> 场景：你需要开发"批量节点操作"功能
> 工具组合：Hermes（总指挥）+ Claude Code（开发）

### 第 1 步：Hermes 设计架构

```
在 Hermes 中输入：

"请用专家团模式，以架构师身份帮我设计'批量节点操作'功能。

需求：
- 用户可以选择多个节点
- 一键执行启动/停止/重启
- 实时显示操作进度
- 支持失败重试

请输出：
1. 系统架构设计
2. API 接口契约
3. 数据模型设计
4. 前后端交互流程"
```

**Hermes 输出示例**：
```
## 架构方案

### 涉及模块
- Server: 新增 BatchOperateController、BatchOperateService
- Agent: 接收批量指令，并行执行
- Frontend: 新增 BatchOperateView.vue

### API 设计
POST /api/batch/operate    批量操作
GET  /api/batch/status/{id} 查询进度

### 数据模型
batch_task 表：
- id: BIGINT PRIMARY KEY
- node_ids: VARCHAR(500) 节点 ID 列表
- operation: VARCHAR(20) 操作类型（start/stop/restart）
- status: VARCHAR(20) 状态
- created_at: DATETIME

### 接口契约
输入: { node_ids: number[], operation: string }
输出: Result<BatchTaskModel>
```

### 第 2 步：Claude Code 实现代码

```
在 Claude Code 中输入：

"请根据以下架构方案，实现批量节点操作功能。

[粘贴 Hermes 输出的完整方案]

请输出：
1. Controller 代码（Spring Boot）
2. Service 代码（业务逻辑）
3. MyBatis Mapper（数据访问）
4. Vue 页面（前端交互）"
```

**Claude Code 输出示例**：
```java
// BatchOperateController.java
@RestController
@RequestMapping("/api/batch")
public class BatchOperateController {
    
    @Autowired
    private BatchOperateService batchOperateService;
    
    @PostMapping("/operate")
    public Result<BatchTaskModel> operate(@RequestBody OperateRequest request) {
        // 参数校验
        if (request.getNodeIds() == null || request.getNodeIds().isEmpty()) {
            return Result.error("节点列表不能为空");
        }
        
        // 执行批量操作
        BatchTaskModel task = batchOperateService.batchOperate(request);
        return Result.success(task);
    }
}
```

```vue
<!-- BatchOperateView.vue -->
<template>
  <div>
    <a-table :data="nodes" @select="handleSelect">
      <a-column title="节点名称" data-index="name" />
      <a-column title="状态" data-index="status" />
    </a-table>
    
    <a-space>
      <a-button @click="handleStart">启动</a-button>
      <a-button @click="handleStop">停止</a-button>
      <a-button @click="handleRestart">重启</a-button>
    </a-space>
    
    <a-progress :percent="progress" />
  </div>
</template>
```

### 第 3 步：验收

```
你：
- 检查 Java 代码是否符合 Spring Boot 规范
- 检查 Vue 页面是否符合 Ant Design Vue 规范
- 本地测试 API 接口
- 部署验证"
```

## 6. 核心原理：为什么能跨工具协作？

### 原理图

```
┌─────────────────────────────────────────────────────────────┐
│  跨工具协作的核心：Skill 文件 = 标准化的"角色说明书"          │
│                                                             │
│  Skill 文件内容：                                            │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  YAML 元数据：                                       │   │
│  │  - name: expert-architect                            │   │
│  │  - description: 何时触发                            │   │
│  │                                                     │   │
│  │  Markdown 正文：                                     │   │
│  │  - 角色定义                                          │   │
│  │  - 背景知识                                          │   │
│  │  - 工作流                                            │   │
│  │  - 输出格式                                          │   │
│  │  - 协作规则                                          │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  复制到其他工具：                                            │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                 │
│  │ Hermes   │  │ Codex    │  │ Claude   │                 │
│  │ 读取     │  │ 读取     │  │ 读取     │                 │
│  │ Skill    │  │ Skill    │  │ Skill    │                 │
│  └──────────┘  └──────────┘  └──────────┘                 │
│       ↓              ↓              ↓                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  每个工具都能理解"专家角色"，按相同规则执行           │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

### 为什么有效？

```
1. 所有 AI 模型都能理解"角色"概念
   → 架构师/后端/前端，是人类通用的角色划分

2. Skill 文件是标准化的 Markdown 格式
   → 所有工具都能读取和解析

3. 核心是"提示词工程"
   → 通过结构化提示词，引导 AI 按特定角色工作
```

## 8. 故障排查

### 问题 1：Skill 文件没有被加载

```
症状：AI 没有按专家角色工作

排查步骤：
1. 检查目录是否正确
   ❌ 错误：skills/expert-architect/SKILL.md
   ✅ 正确：.claude-code/skills/expert-architect/SKILL.md

2. 检查文件名是否正确
   ❌ 错误：skill.md / SKILL.txt
   ✅ 正确：SKILL.md（大写）

3. 检查 YAML 元数据
   ❌ 错误：没有 name 字段
   ✅ 正确：name: expert-architect

4. 重启工具
   → 退出工具，重新打开"
```

### 问题 2：AI 没有按角色输出

```
症状：AI 输出的内容不符合角色定位

解决方案：
1. 在提示词中强调角色
   "请以后端专家身份，输出 Java 代码"

2. 检查 Skill 文件中的"角色定义"部分
   → 确保清晰描述了职责边界

3. 提供具体示例
   "请输出 Controller/Service/Mapper 三层代码"
```

### 问题 3：跨工具交接时信息丢失

```
症状：下一环节拿不到完整的上一环节输出

解决方案：
1. 使用"交接模板"（见第 5 节）
   → 明确说明输入/任务/输出要求

2. 复制完整输出
   → 不要省略关键信息

3. 验证关键信息
   → 检查 API 契约、数据模型是否完整"
```
